# 现有 20 类缺陷与 YOLOE 视觉提示通路融合分析

## 1. 问题定义

当前仓库里实际上有两条能力路径：

- 路径 A：基于现有数据集训练好的 20 类缺陷检测。
- 路径 B：基于 YOLOE visual prompt 的新缺陷检测，通过参考框或掩码生成 visual prompt embedding，再检测数据集中没有的新目标。

目标不是二选一，而是让系统在一次推理里同时具备：

- 对已有 20 类缺陷的稳定检测能力。
- 对新缺陷的 promptable 检测能力。

你提出的两种思路分别是：

1. 先做 visual prompt 相似度判断，不够像时再走原始分类头，并接一个辅助分类网络。
2. 为旧类别准备稳定 embedding，与 visual prompt embedding 拼接后，统一做相似度分类。


## 2. 当前代码事实

先把现状说清楚，否则后面的方案判断会失真。

### 2.1 现有两条路径底层并不是两套完全异构模型

`trainE.py` 和 `trainE_VP.py` 都是从 `YOLOE("yoloe-26s.yaml")` 起步。

- `trainE.py` 用 `YOLOETrainerFromScratch` 训练已有类别。
- `trainE_VP.py` 先加载 `./runs/detect/ybj0310/weights/best.pt`，再冻结几乎所有层，只保留 `savpe` 相关参数训练。

这说明目前的 VP 模型不是另起一套检测器，而是在已有闭集能力之上补了一条 visual prompt 编码能力。

### 2.2 YOLOE 本身已经具备“统一原型空间”的结构

从 `ultralytics/nn/tasks.py` 的 `YOLOEModel.get_cls_pe()` 可以看到：

- 有文本原型时，加入 `tpe`。
- 有视觉原型时，加入 `vpe`。
- 最后直接 `torch.cat(..., dim=1)`，把它们拼成统一的 class prototype bank。

从 `ultralytics/nn/modules/head.py` 的 `YOLOEDetect.forward_head()` 可以看到：

- 检测头并不是传统固定分类器，而是用特征和类原型做相似度分类。
- `self.nc` 会跟随传入的 prototype 数量变化。

这意味着你的第二种思路并不是“重新发明一套逻辑”，而是和当前 YOLOE 的核心设计完全同向。

### 2.3 当前真正阻碍“旧类 + 新类同时检测”的位置不在 backbone，而在推理装配

`ultralytics/models/yolo/model.py` 里，`YOLOE.predict()` 一旦传入 `visual_prompts`，当前逻辑会：

- 根据 prompt 数量重写 `self.model.model[-1].nc`
- 把 `self.model.names` 改成 `object0/object1/...`
- 提取 `vpe`
- 调 `self.model.set_classes(self.model.names, vpe)`

结果就是：VP 模式下旧类别集合被覆盖，只剩下 prompt 类别。

所以当前问题不是：

- 检测头不支持混合
- backbone 不支持混合

而是：

- 推理入口在 VP 模式下把旧类集合替换掉了

### 2.4 当前训练也还没有真正做“旧类 + 新类联合训练”

`ultralytics/nn/tasks.py` 的 `YOLOEModel.loss()` 里，当前仍是二选一：

- 有 `visuals` 时走 visual-prompt loss
- 否则走 text/closed-set 路径

再结合 `trainE_VP.py` 只训练 `savpe`，可以判断出：

- 现阶段更适合先做“推理级融合”
- 真正的“训练级统一”需要后续再补

### 2.5 代码库里已经有一个短期工程兜底方案

根目录的 `combine.py` 已经实现了：

- 标准模型跑已有缺陷
- VP 模型跑新缺陷
- 最后在结果层合并 boxes 和类别 ID

它不是最终的模型级融合，但很适合当短期上线基线。

### 2.6 还有两个现实问题要先对齐

第一，`trainE_hybrid.py` 里写的是 `base_nc=10`，但你的需求描述是已有 20 类缺陷，这里显然没对齐。

第二，`ultralytics/cfg/datasets/ybj20260303-yoloe.yaml` 虽然前 20 个是布料缺陷，但文件里总共列了 80 个名字，后面大量是补位类。后续如果要做统一 prototype bank，必须明确：

- 真正的旧类到底是 20 类还是别的数量
- 参与融合的旧类名字列表到底取哪一部分


## 3. 方案一分析：相似度阈值分流，再回退到原始分类头

## 3.1 方案定义

该方案可以概括成：

1. 候选目标特征先和 visual prompt embedding 算相似度。
2. 如果高于阈值，判为新类。
3. 如果低于阈值，回退到原始分类头。
4. 再由辅助网络输出旧类别。

本质上这是一个：

- `VP 分支`
- `旧类分类分支`
- `阈值门控`

组成的双分支系统。

## 3.2 可行性

理论上可行，但不顺着当前 YOLOE 的主干设计走。

当前 YOLOE 的核心是“统一 prototype bank + 相似度分类”。方案一则是在统一结构之外，再人为增加一个 gate，把分类流程拆成两段。因此它更像研究性分流方案，不像适合长期维护的主线方案。

## 3.3 优点

- 逻辑直观，容易解释给业务方。
- 如果你非常想保留原有旧类头部，不愿意动它，这个方案确实能做到“新类先判，旧类兜底”。
- 阈值是显式控制项，便于人工调节。

## 3.4 主要问题

### 问题 1：阈值会成为系统最脆弱的点

这个阈值会同时受到以下因素影响：

- 提示框或掩码质量
- 新缺陷和旧缺陷的外观相似性
- 光照、纹理、尺度变化
- prompt 图和目标图之间的域偏移
- embedding 标定是否稳定

结果是：

- 阈值低了，很多旧类会被误吸进 VP 分支。
- 阈值高了，真正的新类又会被回退到旧类头。

这类门控往往不是调一个超参就能长期稳定的。

### 问题 2：你会得到两个不同分数空间

一旦新类来自 prompt 相似度，旧类来自原始分类头，就会出现两个不同来源的分数：

- VP 相似度分数
- 旧类分类分数

后续会立刻出现工程问题：

- 这两个分数怎么比较？
- 排序怎么统一？
- NMS 前分流还是 NMS 后分流？
- 同一个框在两条支路都高分时如何去重？

这比“单头统一分类”复杂很多。

### 问题 3：训练定义会明显变复杂

你不只是多加一个头，而是同时引入了三件事：

- 是否进入 VP 分支的门控任务
- 旧类辅助分类任务
- 跨分支的一致性约束

同时，当前 `trainE_VP.py` 只训练 `savpe`，并没有为“门控 + 辅助旧类头”准备现成训练流。所以要落地这个方案，几乎一定要改：

- 检测头
- loss
- predictor
- validator
- 后处理/NMS
- 训练数据流

### 问题 4：它和当前代码主线不一致

当前仓库的主线是：

- 用统一类原型做相似度分类

方案一相当于把它重新拆回：

- 先判断是不是 prompt 类
- 再走旧类分类

这和现有 `YOLOEModel.get_cls_pe()`、`YOLOEDetect.forward_head()` 的思路是逆向的。

## 3.5 实现难度评估

如果按工程改造量评估，方案一属于中高到高难度。

粗略打分：

- 可行性：`6/10`
- 训练稳定性：`4/10`
- 实现难度：`8/10`
- 维护成本：`8/10`

## 3.6 结论

方案一更适合做实验性验证，不适合作为当前仓库的主落地方向。

它不是完全做不出来，而是做出来之后，阈值、跨分支标定和后处理会长期消耗你。


## 4. 方案二分析：旧类原型与 visual prompt 原型拼接，统一相似度分类

## 4.1 方案定义

这个方案可以写成统一形式：

```text
region_feature @ class_prototypes^T
```

其中：

```text
class_prototypes = [old_class_embeddings ; novel_visual_prompt_embeddings]
```

也就是：

1. 为旧类别准备一组稳定 prototype。
2. 为新类别从 visual prompt 生成 prototype。
3. 把两者拼成统一 prototype bank。
4. 用同一个检测头、同一个分数空间，同时输出旧类和新类。

## 4.2 可行性

这是当前最可行、也最符合现有代码结构的方案。

原因很直接：

- `YOLOEModel.get_cls_pe()` 已经支持 `tpe + vpe` 拼接。
- `YOLOEDetect` 本身就是 prototype similarity 分类头。
- 当前问题只是在 `YOLOE.predict()` 的 VP 分支把旧类覆盖掉了。
- `trainE_VP.py` 的训练方式也说明 VP 是在已有检测能力上补 visual prompt 表达，而不是替换闭集能力。

所以方案二不是“额外发明一套模型”，而是“把当前代码已经具备的统一原型能力真正用起来”。

## 4.3 优点

### 优点 1：只有一个分类空间

旧类和新类都在同一个 prototype bank 里竞争，直接带来三个好处：

- 分数可直接比较
- 不需要单独门控阈值
- 后处理只需要一套 NMS 逻辑

### 优点 2：和 YOLOE 原始设计完全同向

YOLOE 的设计本来就是“用类原型替代固定分类器”。因此把旧类原型和 visual prompt 原型放在一起，是顺着模型设计走，而不是逆着走。

### 优点 3：对新缺陷增量扩展友好

后续新增缺陷时，不必重构旧类头，只需要：

1. 准备参考框或掩码
2. 生成新类 visual embedding
3. 追加到 prototype bank

这非常适合“小样本快速上新”的场景。

### 优点 4：可以先做推理级融合，再决定是否补训练级融合

这个方案天然支持分阶段落地：

- 第一阶段：先做推理时的 prototype 拼接，不改训练
- 第二阶段：为旧类准备更好的 prototype
- 第三阶段：如果确实有必要，再做联合训练

这样风险更可控。

## 4.4 真正需要解决的难点

### 难点 1：旧类 prototype 到底用什么

这不是二选一，而是三档递进：

第一档，直接使用文本 embedding。

- 优点：实现最简单。
- 缺点：对布料缺陷这种细粒度视觉类别，文本原型可能不够贴数据域。

第二档，离线统计旧类视觉 prototype。

- 用已有 20 类训练集跑特征提取或 VPE/分类特征聚合。
- 每类取均值或聚类中心。
- 比纯文本更贴近数据分布。

第三档，把旧类 prototype 做成可训练参数，做联合优化。

- 灵活性最高。
- 但训练与实现复杂度也最高。

如果目标是先落地，我建议优先级是：

`文本 prototype -> 离线视觉 prototype -> 可训练 prototype`

### 难点 2：旧类数和槽位设计必须严格对齐

如果要做统一 prototype bank，必须明确三件事：

- `base_nc` 到底是多少，当前业务描述应是 20
- 旧类的真实名字列表是什么
- 新类 prototype 在 bank 中占哪些槽位

当前 `trainE_hybrid.py` 里写 `base_nc=10`，而且源码里还没有真正的 `YOLOEHybridTrainer` 实现，这说明“混合训练”在仓库中还没有闭环，不能当成现成功能来依赖。

### 难点 3：NMS 策略要重新考虑

`YOLOE.predict()` 当前默认会把 `agnostic_nms` 设为 `True`。

如果将旧类和新类统一到一个模型里，这个默认值未必合适。因为 class-agnostic NMS 容易让相互接近的新旧类框互相压制。

更合理的做法通常是：

- 混合模式下把 NMS 改成可配置
- 默认优先考虑 class-aware NMS，或者至少做专项对比

### 难点 4：旧类与新类的分数校准仍需观察

虽然方案二没有方案一那种“跨分支分数空间”的问题，但依然要关注：

- 文本 prototype 是否系统性偏高或偏低
- visual prompt prototype 是否对参考样本过拟合
- 与旧类外观相近的新类是否会互相干扰

不过这是统一分类空间内部的校准问题，比方案一的跨分支标定简单得多。

## 4.5 实现难度评估

如果只做推理级融合，也就是：

- 旧类固定 prototype
- 新类 VPE 追加
- 一次前向同时输出旧类和新类

那么当前代码已经非常接近，属于中低难度。

如果进一步做联合训练，则属于中等难度，但依然明显低于方案一。

粗略打分：

- 可行性：`9/10`
- 训练稳定性：`7/10`
- 实现难度：`5/10`
- 维护成本：`4/10`

## 4.6 结论

方案二是当前最推荐的主方案。

它最大的优势不是“理论上更优”，而是：

- 与现有代码结构一致
- 与现有模型设计一致
- 与当前业务的增量缺陷需求一致


## 5. 两种方案的直接对比

| 维度 | 方案一：阈值分流 + 辅助分类头 | 方案二：统一 prototype bank |
|---|---|---|
| 是否符合当前 YOLOE 结构 | 否 | 是 |
| 是否只有一个分数空间 | 否 | 是 |
| 是否依赖阈值门控 | 是 | 否 |
| 推理后处理复杂度 | 高 | 低 |
| 训练改造量 | 高 | 中低到中 |
| 维护成本 | 高 | 低 |
| 短期能否先落地 | 较难 | 可以 |
| 长期是否适合作为主线 | 不推荐 | 推荐 |


## 6. 推荐落地路线

## 6.1 短期工程路线

如果目标是尽快上线“已有缺陷 + 新缺陷同时检测”，短期可以继续保留 `combine.py` 作为工程兜底方案。

优点是：

- 不改底层模型
- 回归风险最低
- 可以立刻给业务可用结果

缺点是：

- 两次前向
- 分数空间不统一
- 长期维护成本高

所以它适合作为过渡方案，不适合作为最终架构。

## 6.2 中期主线方案

主线建议直接做方案二，也就是改单模型推理装配逻辑。

最小闭环改造建议是：

1. 修改 `ultralytics/models/yolo/model.py` 中 `YOLOE.predict()` 的 VP 分支。
2. 不再在 VP 模式下覆盖旧类 `names/nc`。
3. 保留旧类名字列表，例如 20 个真实缺陷名。
4. 先取旧类 prototype。
   当前最简单做法是旧类文本 embedding。
5. 再通过 `predictor.get_vpe(refer_image)` 提取新类 VPE。
6. 执行：

```python
all_names = base_names + vp_names
all_pe = torch.cat([base_pe, vpe], dim=1)
self.model.set_classes(all_names, all_pe)
```

7. 将混合模式下的 NMS 改成可配置，并重点验证新旧类相邻框的抑制行为。

这一版完成后，就已经能用单模型实现“旧类 + 新类”的联合推理。

## 6.3 后续增强路线

如果推理级融合效果可接受，再往下做两步增强：

第一步，离线构建旧类视觉 prototype，替换纯文本 prototype。

第二步，如果确实需要进一步提升统一空间的一致性，再补真正的 hybrid trainer、hybrid validator 和对应 loss，把旧类与新类一起纳入训练。


## 7. 最终建议

结论很明确：

- 不建议把主精力放在“阈值分流 + 回退旧类头”的方案一上。
- 建议把主精力放在“旧类 prototype + visual prompt prototype 统一拼接”的方案二上。

更具体地说：

- 短期：继续用 `combine.py` 作为工程兜底。
- 中期：改单模型 `YOLOE.predict()`，做统一 prototype bank 推理。
- 长期：如果业务确认需要更强一致性，再补联合训练。

从当前仓库结构、改造量、风险和后续维护成本综合看，方案二明显更值得投入。
